# 第一部分：现状与问题

## 1.1 当前流水线全景

```

  │ 上下文：前幕摘要                                              │

  └──────────────────────────┬──────────────────────────────────┘
                             ▼
  ┌─────────────────────────────────────────────────────────────┐

  │ 分辨率：幕级 → 章级                                          │

  │ 上下文：前 10 章骨架（角色行动/伏笔处理/情绪曲线）              │
  │ 产物：outline.json                                           │
  └──────────────────────────┬──────────────────────────────────┘
                             ▼
  ┌─────────────────────────────────────────────────────────────┐
  │ 逐章扩写（问题区域）                                          │
  │                                                             │
  │  逐场景扩写 → 规则拼装 → 状态卡 → 追踪器更新                  │
  │                                                             │
  │  组件：SceneExpander / SceneAssembler / StateCard            │
  │       CharacterTracker / ForeshadowingTracker                │
  │       EmotionalArcTracker / SatisfactionEngine              │
  └─────────────────────────────────────────────────────────────┘
```

## 1.2 核心问题

| 问题 | 现象 | 根因 |
|------|------|------|
| 章节生硬 | 场景间过渡靠关键词匹配 | 逐场景独立扩写 + 规则拼装 |
| 连贯性差 | 角色位置/伏笔状态不连贯 | 三个 Tracker 的 `get_context_for_chapter()` 全代码库零调用——只写不读 |
| 上下文断裂 | previous_context / after_context 构建了但丢弃 | `_generate_state_card()` 接受参数但不用；`SceneExpander` 不接收任何上下文 |
| 爽点脱节 | 爽点标记从未注入 prompt | `SatisfactionEngine.generate_satisfaction_prompt()` 零调用者 |
| 状态不连续 | 状态卡每章从零分析 | AI 只看到当前章最后 8000 字，无上章状态 |
| 调用浪费 | 每章 4~5 次 API 调用 | 场景数×1 + 状态卡×1，且上下文都相同 |
| 文档脱节 | CLAUDE.md 描述三角色流程 | 代码中不存在 Generator→Refiner→Reviewer |

## 1.3 根因分析

**上下文窗口受限时代的补救架构。** 逐场景扩写是因为当时模型窗口装不下一整章；Tracker 是因为大纲不够详细需要运行时追踪。现在这两个前提都不成立了——大纲已经详细到场景节拍级别，模型窗口也足够装下一整章。

# 第二部分：优化方案（完整业务线）

## 2.1 设计原则

1. **大纲即追踪器。** Stage 2 产出的骨架已含每章的核心事件、因果链、场景概览、角色行动、伏笔处理、情绪曲线——不需要运行时再追踪一遍。
2. **章节级扩写，不拆场景。** 一整章的大纲 + 丰富的上下文 → 一次性生成完整章节。AI 自然处理场景间过渡和节奏变化。
3. **爽点融入幕级设计。** 在幕级规划 prompt 中注入网文节奏原则，让剧情自然承载燃点爆点，而非逐章硬标记。
4. **上下文用原文。** 30章大纲 + 10章正文全文，直接作为 prompt 上下文，不做摘要不截断。

### 数据流对比

```
优化前：
  outline → 拆场景 → SceneExpander(无上下文) × 3~4 次
  → SceneAssembler(规则拼装) → StateCard(AI) → Trackers(只写不读)

  问题：AI调用多、拼装生硬、上下文缺失

优化后：
  骨架 + 30章大纲上下文 + 10章正文上下文 → AI(一次调用) → 完整章节

  特点：AI调用少、过渡自然、上下文丰富，每章 API 调用降低 75-80%
```

### 各维度提升

| 维度 | 优化前 | 优化后 |
|------|--------|--------|
| **章节一致性** | 逐场景独立，无上下文 | 30章大纲+10章正文作为上下文 |
| **网文可读性** | 规则拼装过渡生硬 | 章节级一次生成，AI 自然处理过渡 |
| **逻辑连贯性** | 状态卡每章从零分析 | 前10章正文全文 = 最完整的"状态" |
| **管理方便** | 6个运行时组件 | 1个运行时组件（ChapterExpander） |

## 2.3 上下文窗口设计

### 大纲生成阶段

| 阶段 | 前向上下文 | 说明 |
|------|-----------|------|
| Stage 1 幕规划 | 前幕摘要(1段) | 字符串拼接 |
| Stage 2 章骨架 | 前章核心事件+结尾卡点 | 因果链传递，逐章增量生成 |

### 正文扩写阶段

```
对第 N 章进行扩写时：

│   · 章节标题 / 核心事件 / 场景概览 / 因果链                   │
```

### 滑动窗口示例

```
生成到第 50 章时：

  前30章大纲窗口: 第20~49章的大纲
  前10章正文窗口: 第40~49章的正文全文

  如果用户修改了第35章正文：
    → 第36~45章的正文上下文包含旧版第35章 → 变脏 → 需级联重生成
    → 第46章起：旧版第35章已滑出10章正文窗口 → 不受影响
```

## 2.4 幕级节奏设计

不在章节级别硬性标记"第X章是打脸爽点"，而是在幕级规划时注入网文节奏原则。

### 设计理念

- **情绪曲线规律**：打压→积蓄→反弹→爆发，读者需要呼吸感
- **高潮密度**：每10-15章一个小高潮，每30-50章一个大高潮
- **爽点本质**：不是独立事件，是"读者期待被满足的瞬间"——打脸、升级、揭秘、感情进展、阵营反转
- **自然融入**：每个爽点需要至少 3-5 章铺垫，否则爽感打折

### 幕级 prompt 注入内容

```
【网文节奏设计原则 - 请在幕规划中自然融入以下原则】

1. 情绪曲线控制
   - 每幕应有完整的情绪弧线：建立→压抑→转折→爆发→余韵
   - 避免连续3章以上同质情绪
   - 高潮后必须有缓冲章

2. 爽点自然分布
   - 打脸爽点：反派嚣张→主角隐忍→反转→公开打脸
   - 升级爽点：瓶颈→奇遇/突破→实力跃升→小试身手
   - 揭秘爽点：悬念铺垫→线索积累→真相揭露→格局改变
   - 感情爽点：误会/分离→各自成长→重逢/和解

3. 节奏密度建议
   - 小爽点：每5-8章一次 / 中爽点：每15-20章一次 / 大爽点：每幕末尾
   - 铺垫章占60-70%，爆发章占30-40%

4. 读者情绪管理
   - 不要让读者连续3章看不到希望
   - 每次打压后给一个小奖励
   - 主角可以输但不能窝囊，可以忍但不能无底线
```

作为 Stage 1 幕规划的 prompt 前缀注入，AI 在设计每幕结构时自然融入。

## 2.5 分辨率与 API 调用速查

| 阶段 | 分辨率 | 输入 | 输出 | 上下文 | API 调用 |
|------|--------|------|------|--------|---------|
| Stage 1 幕规划 | 全书→幕 | 全书设定 | 幕结构(含节奏设计) | 前幕摘要 | 每幕 1 次 |
| Stage 2 章骨架 | 幕→章 | 幕结构 | 每章骨架(增强版，含因果链/场景概览) | 前章核心事件+结尾卡点 | 每章 1 次 |
| **章节扩写** | **骨架→正文** | **章骨架** | **完整章正文** | **30章大纲+10章正文** | **每章 1 次** |

```
优化前每章 API 调用：场景数×1 + 状态卡×1 ≈ 4~5 次
优化后每章 API 调用：1 次
降低 75-80%，且单次 prompt 上下文更丰富。
```

## 2.6 生成业务逻辑

### 各阶段的生成单元与依赖

| 阶段 | 生成单元 | 上下文窗口 | 依赖方向 | 修改一个单元的影响 |
|------|---------|-----------|---------|-----------------|
| Stage 1 幕规划 | **全书** | 前幕摘要 | 仅前向 | Stage 2 全书失效 |
| Stage 2 章骨架 | 每章 | 前章因果链 | 仅前向 | 扩写 第 N~N+10 章变脏 |
| 章节扩写 | 每章 | 30章大纲+10章正文 | 仅前向 | 第 N+1~N+10 章变脏 |

### 幕级为什么没有一致性问题

Stage 1 一次规划全书结构，不存在"只生成第2幕"的操作。上下文仅一段前幕摘要，3 次 API 调用成本很低，每次调整直接重新生成全部幕即可。

真正的一致性问题存在于 Stage 2 和正文扩写——它们都是逐章增量生成，每章依赖前文做上下文。

### 核心概念：上下文窗口与"脏"传播

```
以正文扩写为例（上下文窗口 = 10 章正文）：

  已生成： [1][2][3]...[11][12][13][14][15][16]...[20]
  
  用户重生成第12章 →
  
  第13章：上下文 [3..12]，包含旧版第12章 → 脏
  第14章：上下文 [4..13]，包含旧版第13章 → 脏（链式）
  ...
  第22章：上下文 [12..21]，如果12-21都已重生成 → 干净
  第23章：上下文 [13..22]，旧版第12章滑出窗口 → 干净

性质：
  · 脏传播单向（只向后，不影响前面的章节）
  · 脏传播有界（超出上下文窗口后自动截断）
  · 修改第N章 → 级联重生成 N+1 到 N+窗口大小
```

### 四种典型场景

#### 场景 A：全新生成（"生成第1-20章"）

```
顺序生成，每章上下文自然指向最新版本。
一致性问题：无。
```

#### 场景 B：继续生成（"已有20章，生成21-25章"）

```
前提：第21-25章的骨架（Stage 2）必须已存在，否则先生成骨架。

第21章： 大纲上下文[1..21] + 正文上下文[第11..20章]（从磁盘读取）
第22章： 大纲上下文[2..22] + 正文上下文[第12..21章]（21章刚生成）
...

一致性问题：无。前20章已在磁盘上，直接被读取使用。
```

#### 场景 C：中间重生成（"已有20章，重生成12-14章"）

```
第11章 ← 上下文不含第12章 → 不受影响
第12-14章 ← 重生成（上下文从磁盘读取前文章节）
第15-22章 ← 上下文包含旧版12-14 → 脏！需级联重生成
第23章起 ← 旧版12-14已滑出10章正文窗口 → 干净

两种策略：
  策略1（严格一致性）：自动级联重生成15-22章
  策略2（最小成本）：只重生成12-14章，15-22章标记为dirty，用户自行判断
```

#### 场景 D：大纲变更后重生成

```
核心设定/总纲变更：
  → Stage 1 全部重生成 → Stage 2 全部重生成 → 正文全部重生成

仅局部骨架微调（如只改第15章骨架）：
  → Stage 1 不变
  → Stage 2：重生成第15章骨架 → 第16-25章正文变脏（上下文窗口覆盖）
  → 正文：重生成第15章 → 级联第16-25章

建议：骨架修改尽量在扩写前完成。进入扩写阶段后只做局部微调。
```

### 级联范围速查

| 修改位置 | Stage 2 级联 | 正文扩写级联 |
|----------|-------------|------------|
| 核心设定/总纲 | 全书 | 全书 |
| 幕规划 | 全书 | 全书 |
| 第N章骨架 | — | N ~ N+10 |
| 第N章正文 | — | N+1 ~ N+10 |

## 2.7 章节状态管理

### session.json 状态记录

```json
{
  "chapters": {
    "1": "clean",
    "2": "clean",
    "...": "...",
    "12": "clean",
    "13": "clean", 
    "14": "clean",
    "15": "dirty",
    "16": "dirty",
    "...": "...",
    "22": "dirty",
    "23": "clean"
  }
}
```

### 状态定义

| 状态 | 含义 | 行为 |
|------|------|------|
| `clean` | 内容与上下文一致 | 无需重生成 |
| `dirty` | 上下文中的前文章节已被修改 | 建议重生成，`continue` 自动从首个 dirty 章开始 |
| `cosmetic` | 仅润色修改（错别字、修辞） | 不触发级联 |

### 最佳实践

1. **大纲阶段**：一次性生成全书大纲，确认满意后再进入扩写。此时修改成本最低。
2. **扩写阶段**：按幕为单位推进，完成一幕的扩写 → 审阅 → 再开下一幕。幕内修改级联到幕尾，不要跨幕修改。
3. **润色修改**：用户修改正文后，系统提问"是否影响剧情走向？"——是则级联，否则仅标记 `cosmetic`。

## 3.4 上下文构建（CLI 侧）

```python
def build_outline_context(outline: Dict, current_ch: int, window: int = 30) -> str:
    """构建前 window 章的大纲上下文（结构化摘要）。"""
    parts = []
    for ch in range(max(1, current_ch - window), current_ch):
        key = f"第{ch}章"
        if key not in outline:
            continue
        ch_data = outline[key]
        parts.append(f"""
--- 第{ch}章: {ch_data.get('标题', '')} ---
核心事件: {ch_data.get('核心事件', '')}
场景: {_summarize_scenes(ch_data)}
角色行动: {ch_data.get('人物行动', '')}
伏笔: {ch_data.get('伏笔处理', '')}
情绪曲线: {ch_data.get('情绪曲线', '')}
结尾卡点: {ch_data.get('结尾卡点', '')}
""")
    return "\n".join(parts)


def build_draft_context(draft_dir: Path, current_ch: int, window: int = 10) -> str:
    """构建前 window 章已生成正文全文。"""
    parts = []
    for ch in range(max(1, current_ch - window), current_ch):
        filepath = draft_dir / f"第{ch:04d}章.txt"
        if filepath.exists():
            content = filepath.read_text(encoding='utf-8')
            parts.append(f"=== 第{ch}章 ===\n{content}\n")
    return "\n".join(parts)
```

## 3.6 总结

```
优化前（6个运行时组件）：
  Outline → SceneExpander(×4) → SceneAssembler → StateCard → Trackers(×3)
                ↑无上下文           ↑规则拼装      ↑无前文     ↑只写不读

优化后（1个运行时组件）：
  骨架 + 30章大纲上下文 + 10章正文上下文 → ChapterExpander(1次AI调用) → 正文

核心变化：
  · 骨架即大纲 — 增强后的章骨架含因果链/场景概览，直接驱动扩写
  · 章节即单元 — 不拆场景不拼装，AI 自然处理过渡
  · 原文即状态 — 前10章全文比结构化摘要准确
  · 幕级即节奏 — 爽点原则融入幕规划，节奏指引在骨架标注
  · 前向级联 — 改第N章→级联到N+窗口，有界传播
  · 可修改传导 — 用户修改自动影响后续，支持 cosmetic 豁免
```

# 第四部分：用户操作模型与 CLI 命令设计

## 4.1 自动化边界

系统的核心问题是：**哪些是机器自动跑的流程，哪些必须人来介入？**

```
用户必须介入（创作决策）：
  · 编写 core_setting.yaml（世界观、人物、伏笔清单）
  · 编写 overall_outline.yaml（幕结构、转折点、总章数）
  · 审阅生成结果，决定是否通过
  · 修改不满意的正文内容
  · 判断修改是"内容变更"还是"润色变更"

系统自动完成（流程化）：
  · Stage 1-3 大纲生成（有素材后一键触发）
  · 上下文窗口构建（从大纲文件和草稿文件自动拼接）
  · 逐章正文扩写（顺序执行，每章一次 API 调用）
  · 脏传播计算（修改章节后自动标记受影响范围）
  · 状态追踪（clean/dirty/cosmetic 自动更新）
  · 断点续传（continue 命令从第一个 dirty 章节开始）
```


### `touch` — 标记章节修改类型

```

为什么需要：
  · 用户直接编辑 txt 文件，系统无法自动感知
```

### `regenerate` — 重生成指定章节

```

    → 跳过交互，直接级联重生成

为什么需要：
```
